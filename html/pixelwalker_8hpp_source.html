41 /*
42  Y-Walkers and X-Walkers
43 
44 
45  The CPU path for PDF rasterization paints pixels into a buffer one scan line at a time,
46  in horizontal runs of varying length. The source and destination pixel buffers used in
47  the painting--which can be quite complex, due to the PDF transparency model--are all
48  associated with instances of RasterYWalker.
49 
50  RasterYWalker (Class Template)
51 
52  A RasterYWalker keeps track of the start of a single pixel buffer's scan line as we march
53  through a rasterization. Every time we advance a scan line all RasterYWalkers are advanced
54  to the next scan line. For some RasterYWalkers this is a no-op because they are simply
55  modeling the result of a computation that is performed once per scan line, re-using the same buffer.
56 
57  GraphicYWalker (Class Template)
58 
59  A GraphicYWalker is simply a (shape, opacity, color) tuple of RasterYWalkers. Frequently
60  the RasterYWalkers within a GraphicYWalker will be using the same buffer in memory with different
61  pixel layouts (e.g., interleaved ARGB). Frequently unneeded channels will be NULL in a GraphicYWalker.
62 
63  X-Walker (Concept)
64 
65  X-Walkers are used to navigate horizontally through a scan line. While RasterYWalker and GraphicYWalker
66  are (template) classes, X-Walker is a concept modeled by various classes. The idea is that
67  completely general (but rare) combinations of inputs can be handled by GenericRasterXWalker model of
68  X-Walker, while the more common combinations of pixel formats can be handled by specialized
69  X-Walkers whose implementations contain lots of compile-time assumptions which allow the
70  compiler to generate more optimal code.
71 
72  If the type "MyXWalker" models the X-Walker concept, then the following methods must be available:
73 
74  // For these ctors the implementation may assume that CanSpecialize() returned true
75 
76  // Construct an x walker pointing to the pixel at the specified x coordinate in yWalker
77  MyXWalker w( const const_RasterYWalker<signal_traits>* yWalker, int_type x );
78 
79  // Construct an x walker pointing at the specified pixel with the specified layout.
80  MyXWalker( const PixelBufferLayout* layout, const signal_type* pixel );
81 
82  // Return true if MyXWalker's implementation is a match for the provided Y-walker or layout
83  bool works = MyXWalker::CanSpecialize( const const_RasterYWalker<signal_traits>* );
84  bool works = MyXWalker::CanSpecialize( const PixelBufferLayout* layout );
85 
86  // Advance the X-Walker to the next pixel. Specialized implementations should be able to
87  // do this using compile-time-determined constants.
88  NextPixel( &w ); // w is a MyXWalker*
89 
90  // Get the signal value for the nth color component from the current pixel
91  signal_type val = GetSignal( &w, n );
92 
93  All of these routines are called deep inside pixel-processing bottlenecks and it is strongly
94  recommended that, if you implement one, you BOOST_FORCEINLINE all these routines, and make what
95  constants you can compile-time.
96 
97  GenericRasterXWalker (Class Template)
98 
99  The fallback model of X-Walker, {const_}GenericRasterXWalker, works on any pixel format, but is
100  as a result slower because the compiler can't optimize anything away.
101 
102  GraphicXWalker (Concept)
103 
104  Models of the GraphicXWalker concept encapsulate the behavior of three models of X-Walker: one
105  each for shape, opacity and color.
106 
107  This is a concept because of the very common case where a single logical "source" of pixel data
108  contains interleaved data across channels--for example, A interleaved with RGB. Pixel data sources
109  with this property can be modeled by the "UnifiedGraphicXWalker" model of the GraphicXWalker
110  concept.
111 */
