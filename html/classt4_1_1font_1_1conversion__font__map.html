Detailed Description

Holds mappings between character code, glyph index and Unicode values for a font. Modifies Unicode values found in the font as necessary such that the mappings between character code and Unicode value are one-to-one. Used when converting PDF document fonts into a form for external use, such as in a browser.

In a PDF font there is a relationship between character codes (i.e., values that appear in a content stream), glyph indexes, and unicode values. Each character has a corresponding glyph and maps to one or more unicode values.

In the general case:

  • A character code can map to multiple Unicode values. For example, a ligature.
  • A given Unicode value or sequence of values can be mapped by multiple character codes. For example, a font that contains two character codes and glyphs for the letter 'r' may map them both to U+0072.

The mapping requirements are more restrictive, however, when creating a new 'cmap' table for a font that originally mapped content-stream character codes to glyphs but will now be mapping Unicode values to glyphs. This scenario arises with FTPDF translation for Liquid Mode. In this case we need a unique one-to-one mapping between character code and Unicode value. In particular:

  • A character code must map to exactly one Unicode value.
  • A given Unicode value must not be used by more than one character code.

This class creates a set of such mappings for a font.

To do so it sometimes must replace a character's Unicode value or values with some different Unicode value. To do this it chooses Unicode values in the Private Use Area (PUA), which has a range of U+E000 to U+F8FF.

For this revised mapping to be useful it requires that it is used both:

  • For text extraction. The Unicode values that appear in the HTML are the ones generated by this class, including the PUA characters.
  • For font conversion. The converted font's internal mapping table must be based on these same values. So when some Unicode value appears in the HTML the correct glyph will appear.

Note that while this approach maintains "Visual Fidelity" in the UI (the correct glyphs all appear), it does not maintain "Content Fidelity" (the text may contain strange-looking PUA values, rather than the original text).
