26 /*
27  About T4_MEMORY_DEBUG_LEVEL
28 
29  T4 provides several different memory debugging levels intended to help track memory errors like
30  buffer over (and under-) runs, use-after-delete, etc. These affect the behavior of both the
31  T4's memory subsystem and the transient_heap.
32 
33  The levels provide different facilities, and every facility provided in a level is provided in all
34  higher-numbered levels. As you might expect the higher this number gets, the more checking you get.
35  The app also runs slower and consumes more memory.
36 
37  The levels do the following:
38 
39  T4_MEMORY_DEBUG_LEVEL == 0 (default for release builds)
40 
41  - No memory debugging is performed. This is the preferred setting for shipping code.
42  - debug_check_no_purge does nothing.
43  - debug_validate_ptr does nothing.
44 
45  T4_MEMORY_DEBUG_LEVEL == 1 (default for debug builds)
46 
47  - All allocated memory is initialized to 0xAA.
48  - All deallocated memory is initialized to 0xBB.
49  - debug_check_no_purge verifies that a thread does not call t4::malloc during its lifetime.
50  - All allocations have 4/8 bytes of preamble (0x11) and postamble (0x22) upon allocation. These
51  are checked for overwriting on deallocation.
52  - During deallocation, 0x33/0x44 are writting into the preamble/postamble memory locations.
53  These values are used to see if we're double-deleting memory.
54  - Active zero-sized mallocs are counted. At shutdown if this number is not zero it is reported as
55  a leak (even though zero-sized allocations do not actually use any memory)
56  - If you call debug_validate_ptr, it will verify that the pointer is an actual result of a previous
57  call to t4::malloc. It will also check the preamble and postamble to look for buffer over-
58  and underruns.
59  - Correct use of transient_snapshot is checked at runtime.
60 
61  T4_MEMORY_DEBUG_LEVEL == 2
62 
63  - All allocations are tracked in a linked list. At shutdown, any unfreed blocks are reported to
64  the console.
65  - Allocations from transient_heap have preambles and postambles that are checked for buffer overruns,
66  just like heap allocations (this is considerably more expensive, hence the higher debug level)
67 
68  T4_MEMORY_DEBUG_LEVEL == 3
69 
70  - At shutdown, all leaks are reported with a complete stacktrace of their allocation.
71 
72  T4_MEMORY_DEBUG_LEVEL is respected even in release builds, although its behavior when detecting a
73  failure is slightly different. If a T4 memory assertion is triggered in a release build, the
74  t4::memory_assert_failed function is called with an error message.
75 
76  T4_MEMORY_DEBUG_LEVEL must match across all T4 clients and T4 itself. Mismatched definitions will lead
77  to ODR violations and undefined behavior.
78 */
79 #ifndef T4_MEMORY_DEBUG_LEVEL
80  #ifdef _DEBUG
81  #define T4_MEMORY_DEBUG_LEVEL 1
82  #else
83  #define T4_MEMORY_DEBUG_LEVEL 0
84  #endif
85 #endif
