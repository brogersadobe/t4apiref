PDF slang – see Watson bug#1252308

This file contains an embedded CFF font (a subset of StoneInformal-Italic) where the charString for glyph ID 1 starts with 32, 7, ... which is the encoding for -107 cmdVLINETO. In other words, it starts with a vertical line (no "moveto" operator). The Type2 Charstring format requires that a valid type2 char string contain one of the moveto operators prior to any line/curve operators. But Type1 charstrings do not have this requirement. In Type1 if the outline starts without an explicit moveto, there is an implicit moveto to the sideBearing point for the glyph. However, Type2 charstrings do not ever encode sideBearing points because "all sidebearings are considered to be zero and hence unencoded" (see Appendix C, description of "endchar" in "The Type2 Charstring Format" – Adobe's Technical Note#5177).

CoolType, ATM, etc... all contain char string parsers that handle both Type2 and Type1 charstrings, and the code ends up silently using a default sideBearing point of (0,0) for any glyph that has no moveto (and no sideBearing) operator. So in permissive builds we do too.
