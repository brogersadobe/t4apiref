Detailed Description

Concept whose models identify the geometric properties of pixels in a model of the surface concept.

The "Surface Traits" concept is used by BlitToSurface to describe at compile time the depth and geometry of an offscreen pixel buffer. It can be used to describe both array-like and non-array-like surface types. BlitToSurface uses the compile-time properties in Surface Traits models to build encoding routines.

There are two kinds of surface traits: array-like and non-array-like. Array-like surfaces have the property that the components in each pixel can be represented by a built-in integral type (e.g., unsigned char), the components are all byte-aligned within a pixel. More specifically, array-like surfaces can be modeled using a PixelLayout.

Non-array-like surfaces specify their component sizes in bits, and the component sizes are allowed to have different sizes. Individual pixels need not be byte aligned; there can be multiple pixels per byte (e.g., 1-bit gray).

It is not currently possible to represent an indexed color table, or a planar surface, using surface traits.

All models of the Surface Traits concept must have the following members:

app_traits

This identifies the application traits associated with the given surface.

Example:


struct MySurfaceTraits
{
    typedef MyApplicationTraits app_traits;

num_components

This identifies the number of components in each pixel.

Example:


    static constexpr size_t num_components = 3;

little_endian

Identifies whether the layout is big-endian or little-endian. This affects the order of the components within the pixel, and the order of the bytes within each component (if applicable).

Example:


    static constexpr bool little_endian = true;

array_like

This should be a member enum which identifies whether or not the surface's pixels are array-like. The value of this enum defines which other properties are required of the surface.

Example:


    enum { array_like = true };

Properties required for Array-Like Surfaces

If array_like is true, the Surface Traits model must have the following members:

signal_traits

This should be a model of the signal_traits concept appropriate to the type of component stored in the surface. For most array-like surfaces, this should be an instantiation of the ByteSignalTraits template.

Example:


    typedef ByteSignalTraits<MyApplicationTraits> signal_traits;

pad_components

This identifies the number of "extra" components at the "front" of the pixel (where the notion of "front" is affected by endianness).

Example:


    enum { pad_pixels = 1 }; // For example, 32-bit RGB is typically bRGB

pixel_size_in_components

This identifies the size of the pixel, in components.

Example:


    static constexpr size_t pixel_size_in_components = 4; // 32-bit RGB pixels are 4 bytes wide

Properties required for Non-Array-Like Surfaces

Non-array-like surfaces require a very different set of properties. They are as follows:

block_type

This type is used to iterate through a scan line of data. A single block must contain an integral number of pixels, whose positions within the block are determined by the remaining fields. A corresponding Surface's base address and scan line stride must be aligned to this type.

Example:


    typedef unsigned short block_type; // 565-RGB

pixels_per_block

Identifies the number of pixels that fit inside a block.

Example:


    static constexpr size_t pixels_per_block = 1; // One 565-RGB pixel in each unsigned short

pad_bits

The number of "wasted" bits in front of the first component. The precise position of these bits is affected by the endianness of the pixel.

Example:


        static constexpr size_t pad_bits = 0; // No pad pixels in a 565-pixel

bits_per_pixel

The number of bits in each pixel. This must be equal to 8 * sizeof( block_type ) / pixels_per_block. It will be removed in a future release, because it is redundant.

Example:


    static constexpr size_t bits_per_pixel = 16;

component_sizes

A type list wrapper that identifies the sizes of each component. This should be an instantiation of the ComponentSizeN template, where N is num_components. The template parameters are the widths in bits of the components.

Example:


    typedef ComponentSize3<5, 6, 5> component_sizes; // size of R, G and B, in order
