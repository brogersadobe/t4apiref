51 /*
52  An efficient class that safely models an integral value with some "special"
53  alternate states. Well, not completely safely, but certainly more safely than
54  the traditional way of setting aside a special constant like -1 for the purpose.
55 
56  Example Usage
57 
58  using namespace t4;
59 
60  // Return the index of the "foo" string, or "special" if not present
61  int_union<size_t> find_index_of_foo( const vector<string>& vec )
62  {
63  auto it = find( begin( vec ), end( vec ), "foo" );
64  if ( it == end( vec ) )
65  return special;
66  return it - begin( vec );
67  }
68 
69  void main( void )
70  {
71  vector<string> v1( { "alpha", "bravo", "foo", "delta" } );
72  int_union<size_t> index = find_index_of_foo( v1 );
73  assert( index == 3 );
74 
75  vector<string> v2( { "alpha", "bravo", "charlie" } );
76  index = find_index_of_foo( v2 );
77  assert( index == special );
78  }

80  Background
81 
82  The most common use of int_union is to model an integral type with "special" values.
83  We've been doing this sort of thing since the dawn of C, where we use a special
84  value like -1 to mean "no value." In these cases we typically don't really care
85  what the special value is, as long as it's not an otherwise valid value for the
86  variable being represented.
87 
88  For example, we might have a "current page" variable whose value is -1 if we're
89  looking at the document in continuous mode, and there is more than one page intersecting
90  the aperture.
91 
92  There are two problems with this approach.
93 
94  The first is that the space of legal values for page numbers is all nonnegative numbers
95  (since traditionally page numbering starts at 0), so we would like to use an unsigned
96  type to represent it. But if we want -1 to be our special value then the type must
97  be signed. In practice this isn't really a problem--we've cut the maximum possible page
98  number in half but really, 2^31 is a perfectly reasonable implementation limit on the
99  number of pages in a document. One place it does cause issues, though, is in writing code
100  that compiles with some of the more strict integer type conversion warnings, which we
101  really like to have enabled when writing 64-bit-safe code.
102 
103  There are of course other ways of solving this problem, for example using a different
104  special value, e.g., std::numeric_limits<size_t>::max(). But nobody wants to type
105  that every time, so you have to declare a special constant.
106 
107  The second, and bigger, problem is that there is little support for detecting when you've
108  failed to check for one of the "special" values, or when a caller has passed in one of these
109  "special" values to a function not expecting it. For example consider the following code
110  written to the idiom described above:
111 
112  int previous_page(int current_page)
113  {
114  return current_page - 1;
115  }
116 
117  The implementer clearly didn't intend to honor the special "-1" convention, and it's not clear
118  what kind of result would make sense for that input. But there's nothing in the function signature
119  that communicates this expectation, and nobody reads comments. Except for you, apparently.
120 
121  It isn't documented anywhere whether or not the input value is allowed to have the
122  special value of -1. It's not unreasonable to simply declare that an unchecked precondition,
123  and maybe add an assertion. But it would be much nicer if we could use the type system to
124  make this clear in the function signature, and that's what int_union is for.
125 
126  Using int_union
127 
128  To solve the above problems, we'd represent page numbers using the most natural type. It
129  makes sense that this be an unsigned type, since negative page numbers don't mean anything.
130  size_t is in practice always the size of a register and is almost always the best choice for
131  ordinal or cardinal quantities:
132 
133  using page_num = size_t;
134 
135  Now let's define a stateless dummy class and global variable that indicates "multiple
136  visible pages":
137 
138  class multiple_pages_t {} multiple_pages;
139 
140  Here's how we define a variable that holds either a page number or the special "multiple_pages"
141  value:
142 
143  int_union<page_num, multiple_pages_t> current_page;
144 
145  We can do a number of interesting things with current_page. We can initialize it with, or assign
146  to it, a size_t:
147 
148  current_page = 23;
149 
150  We can set it to the "multiple visible pages" state:
151 
152  current_page = multiple_pages;
153 
154  We can compare it to integers, or the special multiple_pages constant, and get the results we
155  expect:
156 
157  current_page = 5;
158  assert( current_page == 5 );
159  current_page = multiple_pages;
160  assert( current_page == multiple_pages );
161 
162  We can even pass it into expressions expecting a value of type page_num:
163 
164  extern void paint_page( page_num x );
165 
166  current_page = 2;
167  paint_page( current_page );
168 
169  Note that paint_page takes a page_num and not an int_union, so we expect that it requires a
170  valid page number. But because of the conversion operator, the following code compiles:
171 
172  current_page = special;
173  paint_page( current_page );
174 
175  This code will trigger the int_union's failure mode, because we are attempting to use an int_union
176  in a special, nonintegral state, as an integer.
177 
178  int_union Failure Modes
179 
180  int_union supports all mathematical operations supported by its corresponding integral type, but it
181  is possible to construct a mathematical operation that is valid for the underlying integral type but
182  nevertheless unrepresentable in the int_union. For example:
183 
184  int_union<int> x = std::numeric_limits<int>::max() - 1; // valid and representable
185  ++x; // not representable by int_union<int>. What do we do?
186 
187  It is also possible to attempt to do math on an int_union in a "special" state, i.e.:
188 
189  int_union<int> x = special;
190  ++x; // can't increment "special."
191 
192  The behavior of the ++x above depends on the int_union's failure policy. There are three choices:
193 
194  throw_on_failure: throws an exception in these cases (std::overflow_error, or std::bad_cast)
195  assert_on_failure: in debug builds, triggers an assertion failure. Does nothing in release.
196  silence_on_failure: does nothing on failure.
197 
198  In the latter two policies failure conditions are ignored, and while it's pretty easy to predict what
199  happens, we recommend that these conditions be treated as coding errors and undefined behavior can ensue.
200 
201  The failure policy of an int_union type is specified by the second parameter in the template argument list:
202 
203  int_union<int, throw_on_failure>
204 
205  throw_on_failure is the default failure policy.
206 
207  Multiple Special States
208 
209  int_union supports multiple "special" states. For example if you wanted to model an integer
210  with a special state for "no value" and a separate special state for "multiple values" you
211  might write it like this:
212 
213  class no_value_t {} no_value;
214  class multi_value_t {} multi_value;
215  int_union<int, no_value_t, multi_value_t> x;
216 
217  int_union supports explicit failure policy specification with multiple special states, e.g.:
218 
219  int_union<int, assert_on_failure, no_value_t, multi_value_t>
220 
221  If the second template argument parameter is not a recognized failure policy it is presumed to be
222  a "special" stateless type, and throw_on_failure is used.
223 
224  Default Special State
225 
226  For engineers that can't be bothered to even roll their own stateless "special" class,
227  int_union provides a class named "special_t" which can be used for an integer with a single
228  associated special state. If no template parameters are specified after the integer type
229  this class is used to model the instantiation's sole special state. For example:
230 
231  int_union<int> x = special;
232  assert( x == special );
233 
234  Arithmetic on int_union
235 
236  All of the arithmetic and comparison operators are supported by int_union. Except for the
237  operators that return a reference (e.g., +=), the arithmetic operators all return a plain
238  Integer type. If you attempt to do arithmetic on an int_union in a special state it will
239  throw a std::bad_cast exception, so if an arithmetic function successfully returns it
240  necessarily returns an integral result.
241 
242  int_union doesn't do anything special in its integer operations--if you add a signed integer
243  to an unsigned int_union in an integral state, you should expect the same (possibly undefined)
244  behavior as if you did the same operation with the built-in integral types.
245 
246  int_union's Overhead
247 
248  int_union is the same size as its stored integral type. Depending on the int_union's failure
249  mode, there can be a small performance penalty for using int_union. In most use cases this is
250  negligible, and absolutely worth the additional safety.
251 
252  int_union does nothing when default-initialized, so you can't count on it being initialized to
253  any particular value. In that sense it's just like a regular int.
254 
255  int_union and constexpr
256 
257  int_union and almost all of its methods can be evaluated at compile time, and used in constexpr
258  expressions.
259 
260  Important Notes on Safety
261 
262  The conversion operator that converts an int_union to its native integer type will throw std::bad_cast
263  if it is in a "special" state. This check will occur if you pass an int_union into a function
264  expecting an integer. It will also occur if you attempt to use an int_union in an arithmetic
265  expression.
266 
267  int_union reserves values at the high end of its associated integral type's range and associates
268  these with the "special" values. So in the above example, the multiple_pages "value" is
269  represented by an int_union with an internal stored value of std::numeric_limits<size_t>::max().
270  If an int_union has n "special" states (see below) they are represented by [max() - n + 1 .. max()].
271  Attempting to assign integral values in that range to the int_union will throw a std::overflow_error.
272  In practice this brings the official implementation limit of certain ranges down by a small amount,
273  which may affect how you write certain algorithms, or validate input data. But this is already the
274  case in implementations that use explicit special values like -1.
275 
276  The specific mapping of template parameters to these reserved integral values is an
277  implementation detail and should not be relied upon in client code.
278 */
