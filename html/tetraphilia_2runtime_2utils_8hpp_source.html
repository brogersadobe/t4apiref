255 // TETRAPHILIA_NO_RETURN implementation
256 /*
257  Function decoration indicating that the function does not return
258 
259  Implementers of functions that ALWAYS throw an exception, rather than
260  returning normally, should insert the TETRAPHILIA_NO_RETURN macro immediately before
261  the return type in the function's declaration. This allows implementers of
262  OTHER functions to call this one without having to write any code that
263  follows the invocation of the non-returning function.
264 
265  For example, consider the following "public" function declaration, whose
266  implementation:
267 
268  void ThisFunctionAlwaysThrows( void );
269 
270  And the following bit of client code:
271 
272  int GetFoo( Bar* bar )
273  {
274  if ( bar == 0 )
275  ThisFunctionAlwaysThrows();
276  else
277  return bar->foo;
278  }
279 
280  Most modern compilers will flag GetFoo with a warning like "not all control
281  paths return a value" because they can't tell that ThisFunctionAlwaysThrows()
282  will never return normally, so GetFoo() doesn't need to have ANY code
283  after that call.
284 
285  However, if the compiler is informed of ThisFunctionAlwaysThrows' interesting
286  property, it can intelligently suppress that warning, while still reporting it
287  where it is actually an error. This is what TETRAPHILIA_NO_RETURN is for. Redeclaring
288  ThisFunctionAlwaysThrows thus:
289 
290  TETRAPHILIA_NO_RETURN void ThisFunctionAlwaysThrows( void );
291 
292  will cause the compiler to compile GetFoo successfully without complaint.
293 
294  Note that GetFoo absolutely should NOT have the TETRAPHILIA_NO_RETURN decoration,
295  because it can return successfully (and will, whenever bar is non-NULL).
296 
297  TETRAPHILIA_NO_RETURN is always defined; however, it expands to nothing on compiler
298  platforms where no such facility exists. Implementers on these platforms either
299  have to live with the warnings or add unnecessary extra code after the invocation
300  of non-returning functions.
301 
302 */
