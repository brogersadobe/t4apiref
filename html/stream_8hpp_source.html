31 /*
32  Conversion of a real-valued type (float or double) to a compact string value representable
33  in PDF syntax.
34 
35  This means:
36 
37  1) No scientific notation
38  2) Drop trailing zeros after a decimal point, and the decimal point itself if integral
39  3) Drop the leading zero (except for the value 0 of course)
40  4) Don't generate anything silly like -0
41 
42  Furthermore, we also want to do a reasonable job of preserving precision. In practice, different
43  PDF consumers use different native types to represent parsed floating-point values: float, double
44  and fixed-point representations are all common.
45 
46  These types vary in the space of values they are capable of representing: 16.16 fixed can only go
47  from -32768 to +32767.99997 and there is no utility to producing more than 5 decimal places. At the
48  other end of the spectrum, a double can be as small as 4.9e-324 (permitting subnormal representation)
49  or as large as 1.8e308 (ignoring inf).
50 
51  For the representation of small values, attempting to support (i.e., successfully round-trip) a value
52  like 4.9e-324 is probably not a good idea. The only way to represent this would be:
53 
54  .000...(317 zeroes)...00049
55 
56  which is really not a good use of bytes, especially given that we're trying to represent a value that
57  is probably an artifact of a very very tiny precision error. So, there is a tradeoff to be made here.
58  We've decided to emit a maximum of six decimal places in t4. This means that doubles < .0000005 will
59  round-trip to 0.
60 
61  At the large end of the representational spectrum, there really isn't much we can do. The "correct"
62  representation of 1.8e308 in PDF would be:
63 
64  18000...(301 zeroes)...000
65 
66  Actually, the last ~290 digits of this can really be anything; when read back in as a double those
67  digits won't have any effect on the parsed value.
68 
69  But it's not like we can just truncate, or pin this value to some maximum. We really have only three choices:
70  write an enormous string, pin the value to some reasonable range, or throw an exception.
71 
72  Precision loss is something we all expect and know how to deal with, but changing a number's value by orders
73  of magnitude is surprising and undesirable. On the other hand, failing a save (due to a thrown exception)
74  just because a ridiculously huge floating-point value crept into a computation is worse. So we pin. The
75  question then becomes: to what range should we pin the floating-point values we write out?
76 
77  Integer representation turns out to be the kicker here: PDF permits integers to appear wherever a real
78  is expected, and such a thing might happen as an artifact of the above rules (e.g., 342.0000 -> will get
79  written out as "342"). So the practical thing to do here is to throw an exception if a floating point
80  value falls outside the representation range of an integer, even though both floats and doubles are
81  capable of representing values far outside that range.
82 
83  Now, of course, there are lots of different kinds of integers. Tetraphilia can handle up to 64-bit signed
84  integers correctly, and future versions will support truly arbitrarily sized integers. But we expect that
85  other PDF viewers choose to use 32-bit signed integers, so we'll use that range as the limit of floating-
86  point values that we're willing to write out. Floating-point values falling outside the range [-2^31, 2^31-1]
87  will be pinned to within that range.
88 
89  How do we pin NaN? In the algorithms in Tetraphilia, NaN most usually crops up when you divide 0 by 0 (there are
90  other ways to get NaN from a calculation, but that's the most common). For this case we follow the precedent
91  set by the PDF spec to treat, when necessary, the quotient of 0 and 0 as if it were 0.
92 
93  This does not mean that to_string cannot throw: for example, it might allocate memory. But if system
94  resources are so low that allocations are failing then failing the save is probably the right thing to do.
95 */
96 template<typename Real>
97 std::enable_if_t<std::is_floating_point<Real>::value,
98 t4::string> to_string( Real val, int precision = 6 );
