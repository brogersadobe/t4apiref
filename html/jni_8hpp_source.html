85 /*
86  Traits class that can be specialized for arbitrary C++ types to facilitate Java/C++ data type interchange.
87 
88  The ntype_info is specialized on one C++ type, henceforth NativeType, which should not be cv-qualified or
89  a reference type. ntype_info provides enough information to create java objects from C++ objects and back again.
90 
91  template<typename NativeType, typename = void>
92  struct ntype_info;
93 
94  Specializing ntype_info establishes a relation between a C++ type and a java type. The java type might
95  be a simple scalar, but you can do more complex things. For example on Android we provide a mapping between
96  all models of the integral rectangle concept and the java class android.graphics.Rect.
97 
98  Specializing ntype_info for a type N allows the following:
99 
100  1) Instances of N can be passed as parameters to call_method/call_static_method
101  2) Instances of N can be returned from calls to call_method/call_static_method
102 
103  We use the term "java entity" here instead of "java object" because simple scalar types like "int" and
104  "long long" don't, and shouldn't, compel the creation of java objects. They instead map to jints and jlongs.
105 
106  Suppose we want to bind the following Java class:
107 
108  package com.example;
109  class JPoint
110  {
111  public float x;
112  public float y;
113 
114  JPoint(float x, float y)
115  {
116  this.x = x;
117  this.y = y;
118  }
119  }
120 
121  To the following native class:
122 
123  struct NPoint
124  {
125  float m_x;
126  float m_y;
127 
128  NPoint(float x, float y)
129  : m_x( x ), m_y( y )
130  {}
131  };
132 
133  Declare the specialization:
134 
135  template<>
136  struct ntype_info<NPoint>
137  {
138 
139  Write a static member that returns the JNI class name of the class. Don't forget to use slashes
140  instead of .'s as the delimiter. I swear I make that mistake every single time I write JNI code:
141 
142  static auto get_class_name( void ) { return T4_STR( "com/example/JPoint" ); }
143 
144  Write a static member function that creates a java JPoint from a C++ NPoint:
145 
146  static jobject from_native( JNIEnv* env, NPoint const& r )
147  {
148  return create_java_object( env, get_class_name().c_str(),
149  r.m_x, r.m_y );
150  }
151 
152  Write a static member function that creates a native NPoint from a java JPoint:
153 
154  static NPoint to_native( JNIEnv* env, jobject obj )
155  {
156  check_is_instance_of( env, obj, find_class( env, get_class_name().c_str() ) );
157  return { get_field<jfloat>( env, obj, "x" ), get_field<jfloat>( env, obj, "y" ) };
158  }
159  };
160 
161  and you're done.
162 
163  Note that the C++ type-to-Java-type mapping is many-to-one. Most (but not all) specializations
164  of ntype_info will have a jtype of jobject.
165 
166  The second template parameter is present to allow specializations to use enable_if to define ntype_info
167  for all models of a concept. For example:
168 
169  template<typename Rectangle, std::enable_if_t<typename t4::is_rectangle_type<Rectangle>>::type>
170  struct ntype_info ...
171 
172  Identifies a specialization that will be used for any type that models the Rectangle concept.
173 */
174 template<typename NativeType, typename> struct ntype_info;

212 /*
213  Same as jni::global_ref<jobject>, except that JNIEnv* is obtained on-demand from the current thread and not
214  stored as a member.
215  */
216 template<typename JType>
217 class global_ref2
218 {
219  JType m_obj = nullptr;
220 
221 public:
222  // Default ctor
223  global_ref2( void ) {}
224 
225  // Construct a global_ref from a local reference.
226  global_ref2( JType obj )
227  {
228  JNIEnv* env = GetJNIEnv();
229  if (!env)
230  throw std::runtime_error("");
231 
232  m_obj = static_cast<JType>(env->NewGlobalRef(obj));
233  }
234 
235  // Move ctor
236  global_ref2( global_ref2&& rhs )
237  {
238  boost::swap(m_obj, rhs.m_obj);
239  }
240 
241  global_ref2& operator=( global_ref2&& rhs )
242  {
243  boost::swap( m_obj, rhs.m_obj );
244  return *this;
245  }
246 
247  // dtor
248  ~global_ref2( void )
249  {
250  try
251  {
252  if (m_obj)
253  {
254  JNIEnv* env = GetJNIEnv();
255  if (env)
256  env->DeleteGlobalRef(m_obj);
257  }
258  }
259  catch (...)
260  {
261  }
262  }
263 
264  // Convert to JNI type
265  operator JType( void ) const
266  {
267  return m_obj;
268  }
269 
270 private:
271  // Disabling copy. Copy *can* work, but it is very inefficient because the constructor and destructor must both
272  // attach to the current thread. We could allow this and discourage its usage, but it is too easy to inadvertently
273  // pass by value.
274 
275  // Copy ctor
276  global_ref2( global_ref2 const& rhs ) = delete;
277 
278  // Assignment
279  global_ref2& operator=(global_ref2 const& rhs) = delete;
280 };

287 /*
288  About exceptions in t4::jni
289 
290  C++ and Java both use exceptions to report the failure of an operation, but they're different systems and
291  throwing C++ exceptions out of a JNI method implementation is not allowed. t4::jni provides support for
292  managing exceptions across these boundaries.
293 
294 
295  Exceptions when calling from Java into C++
296 
297  t4::jni::call_cpp (or T4_CALL_JNI_METHOD, which uses call_cpp) should be used inside any JNI function implementation.
298  It takes a JNIEnv* and a functor, executes the functor, and "translates" any thrown exceptions into a Java exception
299  that is then propagated into the caller.
300 
301  call_cpp will attempt to map exceptions from C++ to Java. For example:
302 
303  std::bad_alloc <-> java.lang.OutOfMemoryError
304  std::invalid_argument <-> java.lang.IllegalArgumentException
305  (this list may, and probably should, grow)
306 
307  All other C++ classes get mapped to the com.adobe.t4.NativeException class.
308 
309  Example:
310 
311  extern "C" JNIEXPORT void JNICALL Java_com_example_Foo_OutOfMemoryExample( JNIEnv* env, jobject ths ) BOOST_NOEXCEPT
312  {
313  t4::jni::call_cpp( env, [&]()
314  {
315  // Do something ridiculous to trigger bad_alloc
316  t4::vector<int> my_vector( 0x7FFFFFFF );
317  } );
318  }
319 
320  Note the BOOST_NOEXCEPT attribute; call_cpp will never throw a C++ exception. You should typically _only_ use call_cpp
321  as the only function call in the top level of a JNI entry point.
322 
323  When Foo.OutOfMemoryExample is called, the vector construction will throw std::bad_alloc. call_cpp will
324  catch this exception and map it to java.lang.OutOfMemoryError.
325 
326 
327  Exceptions when calling from C++ into Java
328 
329  t4::jni provides a number of APIs that are simple wrappers around familiar JNI calls, but they report errors
330  via C++ exceptions. The basic idea is this:
331 
332  1) Call the corresponding JNIEnv method
333  2) Call ExceptionOccurred() to see if an exception was installed
334  3) If an exception is pending, throw an internal C++ exception of unspecified type. This type unwinds
335  all the way back to the enclosing call_cpp, insuring that the algorithm does not continue to execute
336  after a failure has occurred.
337 
338  For example:
339 
340  find_class(env, "com.example.this.class.does.not.Exist");
341 
342  Will do three things:
343 
344  1) Call env->FindClass(), which will return nullptr since the class does not exist
345  2) Call env->ExceptionOccurred(), which will return a non-null value indicating that
346  an exception is pending
347  3) Throw a C++ exception that corresponds to the pending Java exception. In this case
348  java_exception containing a jthrowable that is an instance of java.lang.NoClassDefFoundError
349  will be thrown.
350 
351 
352 
353  Throw Clauses
354 
355  You have to be careful with your throws clauses. call_cpp doesn't know what exceptions the declaration
356  of your native method claims to throw. Although C++ supports throws clauses, it's generally agreed that
357  they are a bad idea, and they are now deprecated. They're a bad idea in Java too, but Oracle hasn't gotten
358  the memo yet.
359 
360  So the safest thing to do is simply to declare your native methods that use call_cpp as "throws Exception"
361  (you don't need to write "throws Throwable" because the only other subclass--Error--is implicitly throwable
362  from any Java method). If you don't do this, the exception will get thrown anyway. This is worth emphasizing:
363  Java exception specifications are a compile-time-only feature.
364 */
365 
366 /*
367  Execute C++ code and convert any thrown exceptions to Java exceptions installed in the JNIEnv.
368 
369  Most of the time you won't use this; just write a proper C++ function and invoke it using the
370  T4_CALL_JNI_IMPL macro. But if you wanted to do this manually, you could write:
371 
372  extern "C" JNIEXPORT jobject JNICALL Java_com_adobe_t4_pdf_Document_nativeGetPageGeometry(JNIEnv* env, jclass, jlong doc, jint pageNum) BOOST_NOEXCEPT
373  {
374  return t4::jni::call_cpp( env, [&]()
375  {
376  // If an exception occurs anywhere in this block it will get caught and converted to a Java exception in the JNIEnv.
377 
378  auto doc = t4::jni::get_native_object<t4::pdf::document>( env, doc );
379  auto pg = doc->get_page_geometry( pageNum );
380  return t4::jni::native_to_java_object<t4::pdf::page_geometry>( env, pg );
381  } );
382  }
383 
384  call_cpp will _never_ throw a C++ exception.
385 */
386 template<typename F>
387 std::result_of_t<F()> call_cpp( JNIEnv* env, F&& f ) BOOST_NOEXCEPT;

389 /*
390  C++ exception class that is used to wrap arbitrary jthrowables. If a Java exception is detected
391  in a JNIEnv after a JNIEnv method is called, and that object doesn't have a natural mapping to a
