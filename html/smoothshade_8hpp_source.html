59 // NOTES FOR OPACITY CHANNEL SMOOTH SHADES:
60 // For hobbes/twist we have added a not=ion of "opacity channel gradients", where the
61 // smoothness is pinated into the opacity channel of the raster painter, instead of the
62 // more traditional color channel.
63 //
64 // There are many issues raised here, especially about color spaces and color conversions;
65 // In the future with multi-channel alpha, there may be different conversions and such between
66 // alpha levels in different channels. As such a color-space-ike notion with multi-channel color
67 // conversion might make good sense. Other concepts like background color and clip bbox are also
68 // sensible.
69 //
70 // Since the smooth shade code itself really knows nothing about color, just signals and interpolation
71 // thereof, it is just as happy to interpolate one or more channels of opacity as it is to interpolate
72 // with colors. The only real difference is the resulting raster painter, which should have full
73 // shape/opacity/color channels when doing color shades, but only shape/opacity channels with opacity
74 // shades.
75 //
76 // For now, a simple uncalibrated one channel opacity shade can be accompished by:
77 //
78 // 1) setting SmoothShadeSamplerContext.m_useOpacityChannel to true
79 // 2) setting SmoothShadeSamplerContext.m_dstCSPtr to be a DeviceGray color space
80 // 3) setting SmoothShadeCommonParams.m_colorSpacePtr to the same DeviceGray color space
81 //
82 // WARNING: the notes above imply that a "multichannel" opacity will work. It might, but has not been tried.
83 // if you require this, please be waned it may take us a little while to iron ou the kinks.
84 //
85 // As for the other params, like the "function" params for various smooth shades, it is supposed to be a mapping
86 // from the shades intrinsic data type to the shades source color space (for axial/radial, a mapping from
87 // paramteric "t" to the smooth shades source color space. This should still be true, excpet insofar as the
88 // source "color space" is really an "opacity space". Similar arguments apply for the background color and such.
89 //
90 // I guess a good summary is that replacing the word "color space" with "opacity space" in the PDF spec should
91 // informn the bahavior of rendering when useOpacityChannel is on.
92 //
93 // Note further that in the "useOpacityChannel" case, the resultiung raster painters will have NULL in their
94 // segments' color channels. This is because there is no meaningful olor data stored there; these raster painter
95 // must be used as the "mask" parameters of compositors.
