29 /*
30  Historically T4 has used the "thread local transient heap" for certain classes of allocations for
31  performance reason. Objects allocated on the thread local transient heap have hard-to-describe
32  but useful lifetimes--they will be destroyed when the nearest enclosing transient_snapshot on that
33  thread local heap is destroyed.
34 
35  This practice has the following appealing properties:
36 
37  1) "Large" but locally scoped objects can be allocated "off the stack," which is a good practice in
38  environments with small stack sizes
39  2) Objects allocated this way are reliably LIFO, allowing for a much simpler and faster page-based
40  allocation strategy
41  3) Objects with trivial destruction semantics can be allocated with a simple pointer bump, and the
42  overhead of managing subsequent destruction can be skipped
43 
44  Benefit #1 isn't really a concern any more in modern hardware; it mattered when stacks were ~32K without
45  any sort of VM support. #2 and #3 are still tangible, measurable benefits.
46 
47  This practice (and special allocator techniques in general), however, completely undermines the utility
48  of security/hardening facilities like the Partition Allocator, as it comingles arbitrary objects
49  with data buffers in ways that the Partition Allocator is intended to prevent.
50 
51  So, in environments where the Partition Allocator is intended to be used, we disallow use of the
52  transient_heap (this work is being done in phases and may not be complete as you read this). We use the
53  T4_TRANSIENT_HEAP macro to detect whether or not the above practice is proscribed. So that clients
54  can write more or less the same code regardless of whether or not we're using the thread local
55  transient_heap.
56 
57  These allocations--that are "transient" in a regular build and proper shared pointers in a PartitionAlloc
58  build--are herein called "brief" allocations, and pointers to these allocations are "brief" pointers.
59  They are so called because their lifetimes are expected to be "brief" and scoped, although scoped at the
60  algorithm level and not at the literal C++ stack frame scope level.
61 
62  To create a brief pointer to an object:
63 
64  brief_ptr<foo> bp = t4::make_brief<foo>( ctor_param1, ctor_param2, ... );
65 
66  Parameters to the constructor are forwarded. The return type, brief_ptr<foo>, is just an alias to foo*
67  in the non-PartitionAlloc builds, and an actual class in PartitionAlloc builds. In this second case
68  brief_ptr is really just a wrapper class around a std::shared_ptr (allocated out of the t4 heap) and,
69  as a result, has a longer lifetime then in the non-PartitionAlloc build, so it's important to test both
70  builds (in debug builds out-of-scope misuse of a transiently allocated pointer is quickly found with a
71  crash).
72 
73  To get the raw pointer from a brief_ptr:
74 
75  foo* p = t4::get_ptr( bp );
76 
77  To use "brief" memory with an STL container, use the t4::brief_allocator:
78 
79  std::vector<foo, t4::brief_allocator<foo>> v;
80 
81  Alternatively, many containers have "brief" aliases:
82 
83  brief_vector<foo> v;
84 
85  In this case "brief" memory is simply allocated from the thread local transient_heap in non-PartitionAlloc
86  builds, and from the t4 heap in PartitionAlloc builds.
87 */
