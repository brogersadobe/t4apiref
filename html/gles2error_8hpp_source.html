33 /*
34  Calls glGetError() and throws an exception if it is nonzero. The exception will be one of the
35  kTetRuntime*Error exceptions, because this API is intended to propagate useful information to
36  the user, not the engineer (who presumably can just set a breakpoint on throw and get all the
37  details they need).
38 
39  The mapping from GLenum to tetraphilia::error is this:
40 
41  GL_NO_ERROR -> Returns control to caller without throwing
42  GL_OUT_OF_MEMORY -> std::bad_alloc
43  GL_INVALID_OPERATION -> std::out_of_range
44  GL_INVALID_ENUM, GL_INVALID_OPERATION, GL_INVALID_FRAMEBUFFER_OPERATION ->
45  assertion failure followed by std::out_of_range. This is
46  probably an actual programming error, but it could conceivably be
47  a pathological PDF causing the GPU to exceed some internal, unpublished
48  limit. For example, a file with 10000 nested transparency groups would
49  likely blow out the GPU's framebuffer or texture cache.
50  Everything else -> assertion failure (the above list _should_ be exhaustive)
51  followed by std::runtime_error( "" ).
52 */
53 void CheckGLError( void );

62 #ifdef _DEBUG
63 /*
64  Calls glGetError() and triggers an assertion if the result is nonzero. Has no effect
65  in release builds.
66 
67  The intention with DebugCheckGLError is to allow debug builds to signal GL failures
68  closer to the actual point of error, while allowing release builds to avoid calling
69  glGetError() so often that performance is compromised. This is harder to do than it
70  sounds, because calling glGetError() resets the GL context's current error. So,
71  DebugCheckGLError stashes away the result in a global so that subsequent calls to
72  CheckGLError() will see the error they _would_ have gotten if those other, debug-only
73  calls to glGetError() hadn't happened.
74 
75  DebugCheckGLError() does _not_ throw an exception; it triggers an assertion failure.
76  If you continue program execution you should see an exception thrown on the next call
77  to CheckGLError.
78 
79  In summary:
80 
81  1) Call DebugCheckGLError() a lot. It is a no-op in release builds.
82  2) Call CheckGLError() sparingly, but enough such that GL errors are always
83  eventually detected and handled somehow.
84 */
85 void DebugCheckGLError( void );
86 void DebugCheckBackgroundError( void );
87 #else
88 inline void DebugCheckGLError( void )
89 {}
