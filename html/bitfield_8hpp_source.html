29 /*
30  Utility class for bitfields using an enumerated type.
31 
32  Enumerated types in C++ are frequently misused. Enums (and enum classes) are supposed to be used
33  to enumerate a finite space of options, but the fact that they're implemented as simple integer
34  types leads to all kinds of abuses. Consider the following:
35 
36  enum my_flags
37  {
38  foo = 1 << 0,
39  bar = 1 << 1,
40  baz = 1 << 2
41  };
42 
43  my_flags operator|( my_flags lhs, my_flags rhs )
44  {
45  return static_cast<my_flags>( static_cast<int>( lhs ), static_cast<int>( rhs ) );
46  }
47 
48  The fact that so many static_cast calls are required is a hint that you're doing something bad.
49 
50  This isn't really a valid use of enums. Clearly the author wants to support code like this:
51 
52  my_flags z = foo | baz;
53 
54  But the value of "z" (5) after this expression executes isn't actually a valid member of the my_flags
55  enumeration. Sure, the code all works, but this sort of usage confounds some useful compiler warnings
56  like -Wswitch-enum and -Wcovered-switch-default.
57 
58  On the other hand, what the author of this code wants is perfectly reasonable, and isn't really
59  supported in a straightforward way by the standard library. std::bitset comes close. t4::bitfield
60  supports mapping an enumerated type to a set of bits, one bit per member of the enumeration.
61 
62  Example:
63 
64  // Do NOT assign special values to these; just let the default happen, else you'll get
65  // into trouble
66  enum my_flags
67  {
68  foo,
69  bar,
70  baz
71  };
72 
73  bitfield<my_flags, 3> b = { foo | baz };
74 
75  Unfortunately, because C++ has no enum introspection APIs, it's your job to keep the number of entries
76  in the enumeration in sync with the NumBits template parameter.
77 */

255 /*
256  Macro which defines a relationship between a bitfield type and a flag enum type. This creates the
257  following alias in the current namespace:
258 
259  using bitfield_type = t4::bitfield<flag_type, num_flags>;
260 
261  And also defines the ~, |, & and ^ operators to produce instances of bitfield_type when given operands
262  of type flag_type. This macro should be invoked in the namespace in which flag_type is defined.
263 */
264 #define T4_DEFINE_BITFIELD( bitfield_type, flag_type, num_flags ) \
265  using bitfield_type = t4::bitfield<flag_type, num_flags>; \
266  constexpr inline bitfield_type operator~( flag_type lhs ) { return ~bitfield_type( lhs ); } \
267  constexpr inline bitfield_type operator|( flag_type lhs, flag_type rhs ) { return bitfield_type{ lhs, rhs }; } \
268  constexpr inline bitfield_type operator&( flag_type lhs, flag_type rhs ) { return bitfield_type( lhs ) & rhs; } \
269  constexpr inline bitfield_type operator^( flag_type lhs, flag_type rhs ) { return bitfield_type( lhs ) ^ rhs; }
