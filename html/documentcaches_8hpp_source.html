40 /*
41  Aggregating class for storage of objects relevant to a Store's interpretation
42  as a PDF file.
43 
44  This is more or less the analogue to the PDDoc class in PDModel. The Store
45  class is just a DOM-like object with very little in the way of document-level
46  semantics; the DocumentCaches class is an aggregation of a lot of classes that
47  support the interpretation of an underlying Store as a PDF document.
48 
49  Most of these classes are caches, which provide services for taking a Dictionary
50  and producing a smart_ptr to a derived class such as Font or ColorSpace. It
51  also includes the PageMap class, which manages the mapping to and from PageIDs
52  and PageIndexes (0-based page numbers).
53 
54  Do NOT add any more get_xxx methods to this class. Instead use the technique
55  described below.
56 
57  Adding objects to DocumentCaches
58 
59  DocumentCaches' purpose is simply to provide a scope for objects whose lifetimes
60  should match the containing document. Most of these objects are caches, but not
61  all (e.g., PageMap). The original design was simply to add member variables to
62  this class and corresponding accessors, but that doesn't scale well, and pulling
63  a DocumentCaches into the compile just gets less and less pleasant as the space
64  of these "appendage" objects grows.
65 
66  So now all new additions should simply go through get_appendage. Suppose we
67  write the following class:
68 
69  class foo_cache
70  {
71  DocumentCaches<app_traits>& m_docCaches;
72  int m_depth;
73 
74  public:
75  foo_cache( DocumentCaches<app_traits>& docCaches, int depth );
76 
77  foo get( t4::pdf::dictionary const& d );
78  };
79 
80  And we'd like a single foo_cache of depth 23 to be associated with a document,
81  to exist for the lifetime of the document itself. So we write a public function:
82 
83  foo_cache& get_foo_cache( DocumentCaches<app_traits>& dc )
84  {
85  return dc.get_appendage<foo_cache>( dc, 23 );
86  }
87 
88  (Sometimes you will need to add a "template" between the . and the get_appendage, the
89  compiler will help you out there).
90 
91  Now a client can get a DocumentCache's associated foo_cache by simply calling:
92 
93  foo_cache& fc = get_foo_cache( my_document_caches );
94 
95  Note that no changes were required to the DocumentCaches class to provide this function,
96  and we don't need to add any #includes to this header file.
97 
98  See the implementation of GetRoleMapCache() in taggedstructurebuilder.hpp for a real
99  example of this.
100 */
101 template<typename _app_traits>
102 class DocumentCaches
103 {
104 public:
105  typedef _app_traits app_traits;
106 
107 private:
108  // Document object caches.
109  typedef store::Store<app_traits> store_type;
110 
111 public:
112  DocumentCaches( store_type& store, t4::pdf::document& document )
113  : m_document( document ),
114  m_store( store )
115  {}
116 
117  t4::pdf::document& GetDocument( void ) const { return m_document; }
118 
119  store_type& GetStore()
120  {
121  return m_store;
122  }
123 
124  /*
125  When adding new caches, please use this mechanism instead of adding more methods as above. The arguments
126  are forwarded to the appendage's ctor.
127  */
128  template<typename Key, typename... TArgs>
129  typename t4::appendage_type<Key>::type& get_appendage( TArgs&&... args )
130  {
131  return m_appendages.get<Key>( std::forward<TArgs>( args )... );
132  }
133 
134 private:
135  t4::pdf::document& m_document;
136  t4::appendage_map m_appendages;
137  store::Store<app_traits>& m_store;
138 };
