476 /*
477 Notes on Java UTF-8 handling.
478 
479 JNI provides two approaches for converting Java Strings to/from native strings.
480 
481  GetStringUTFChars() / NewStringUTF()
482  Retrieve Modified UTF-8 from a Java String; create a Java String from Modified UTF-8.
483 
484  GetStringChars() / NewString()
485  Retrieve UTF-16 from a Java String; create a Java String from UTF-16
486 
487 Since GetStringUTFChars()/NewStringUTF() deal with UTF-8, it would seem natural to use these to convert into and out of
488 std::basic_string. This would be wrong.
489 
490 "Modified" UTF-8 is essentially a format otherwise known as CESU-8. For values outside the Unicode Basic Multilingual
491 Plane (BMP), CESU-8 and UTF-8 representations are identical. But for BMP code points (code points above 0xFFFF) they
492 differ. (In addition to using CESU-8 encoding, Modified UTF-8 has special handling for null values -- which is not
493 applicable in thsi discussion.)
494 
495 UTF-8 encodes non-BMP code points in a four-byte sequence. CESU-8 instead represents the code point as a surrogate pair,
496 then encodes each of these as UTF-8. For example, for the character 0x1F608 the encodings look like:
497  UTF-8 F0 9F 98 88
498  CESU-8 ED A0 BD ED B8 88 (0xD83D 0xDE08)
499 
500 Note that a UTF-8 decoder *could* be written so that it detects the surrogate pair and then combines them into a single
501 code point. This would be wrong, according to the UTF-8 spec (RFC3629), which is quite stringent about this point.
502 
503  "(T)he CESU-8 encoding of those characters is NOT valid UTF-8."
504 
505  "Implementations of the decoding algorithm above MUST protect against decoding invalid sequences. For instance, a
506  naive implementation may decode ... the surrogate pair ED A1 8C ED BE B4 into U+233B4. Decoding invalid sequences
507  may have security consequences or cause other problems."
508 
509 It is worth noting that apparently this admonition is frequently ignored by UTF-8 decoders. Note that Android Acrobat
510 uses GetStringUTFChars() to process filenames, and is able to open files containing non-BMP characters in their names.
511 
512 [It seems to me that there is no valid use for GetStringUTFChars(). One of the following will be true:
513  - The surrogate pairs do not get further decoded, in which case the code using the string will fail.
514  - The surrogate pairs do get further decoded, in which case a security risk is introduced.]
515 
516 Because of all this, t4::jni uses GetStringChars() / NewString(). This introduces an extra conversion between UTF-16 and
517 UTF-8.
518 
519 See:
520  https://en.wikipedia.org/wiki/UTF-8#Modified_UTF-8
521  https://en.wikipedia.org/wiki/CESU-8
522  https://docs.oracle.com/javase/1.5.0/docs/guide/jni/spec/types.html#wp16542
523  https://tools.ietf.org/html/rfc3629
524 */
525 
526 template<typename string_type>
527 string_type to_utf8( JNIEnv* env, jstring jstr )
528 {
529  // Extract chracters as UTF-16 and convert to UTF-8. See notes above regarding UTF-8 handling.
530  jni_detail::jstring_chars jc( env, jstr );
531  std::wstring_convert<std::codecvt_utf8_utf16<char16_t>, char16_t, typename string_type::allocator_type::template rebind<char16_t>::other, typename string_type::allocator_type> conv;
532  auto cnv = []( jchar jch ) { return static_cast<char16_t>( jch ); };
533  auto start = boost::make_transform_iterator( jc.begin(), cnv );
534  auto stop = boost::make_transform_iterator( jc.end(), cnv );
535  t4::basic_string<char16_t> wchs;
536  std::copy( start, stop, std::back_inserter( wchs ) );
537 
538  return conv.to_bytes( wchs.data(), wchs.data() + wchs.size() );
539 }

757 /*
758  Very simple work queue built out of boost.Asio components. Callers
759  provide callables that are then scheduled on an asio::io_context.
760  The callers immediately wait on the returned future. All we're really
761  doing here is handing the task off to another thread that is known to
762  be jni-safe, i.e., not in the middle of a coroutine or fiber or some
763  other such construct which is known to confound JNI.
764 */
765 class jni_work_queue
766 {
767  // io_context managing the work requests
768  boost::asio::io_context m_io;
769 
770  // A "work guard" which ensures that the io_context will not automatically
771  // stop the minute its queue is empty.
772  boost::asio::executor_work_guard<boost::asio::io_context::executor_type> m_work;
773 
774  // Guards m_threads in case we decide to support adding additional threads in
775  // the future. This could become necessary if we decide to support scenarios
776  // like calling into JNI from a coroutine, then starting a coroutine, then calling
777  // back into JNI again.
778  t4::mutex m_mutex;
779 
780  // Queue of all active threads available to service JNI calls.
781  std::vector<t4::thread> m_threads;
782 
783 public:

785  jni_work_queue( void );
786 
787  ~jni_work_queue( void );

789  template<typename F>
790  decltype(auto) operator()( F&& f )
791  {
792  spawn_at_least_one_thread();
793  assert( !already_on_jni_work_thread() );
